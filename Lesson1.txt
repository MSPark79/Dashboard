다음 작업을 멀티 에이전트 역할로 나눠보세요.

작업: 에이전트 운영 대시보드 첫 화면 만들기

1. 어떤 역할이 필요한가?
2. 각 역할의 입력과 출력은 무엇인가?
3. 사람의 승인 지점은 어디인가?
4. 로컬에서 검증할 명령은 무엇인가?



기획자, 구현자, 리뷰어, 검증자



1. 필요한 역할
- 프로덕트 매니저 (PM): 요구사항 정의, 사용자 시나리오 설계  
- UX/UI 디자이너: 화면 레이아웃, 인터랙션 플로우, 시각적 디자인  
- 프론트엔드 개발자: 대시보드 UI 구현, 컴포넌트 개발  
- 백엔드 개발자: 데이터 API 설계 및 제공  
- 데이터 엔지니어/애널리스트: 대시보드에 표시할 핵심 지표 정의 및 데이터 파이프라인 구축  
- QA 엔지니어: 기능 및 UI 검증, 버그 리포트  
- DevOps 엔지니어: 로컬 및 배포 환경 설정, 빌드/배포 자동화  


2. 각 역할의 입력과 출력
| 역할 | 입력 | 출력 |
|------|------|------|
| PM | 사용자 요구사항, 비즈니스 목표 | 기능 정의서, 화면 요구사항 문서 |
| UX/UI 디자이너 | 요구사항 문서, 사용자 시나리오 | 와이어프레임, 디자인 시안 |
| 프론트엔드 개발자 | 디자인 시안, API 스펙 | 대시보드 UI 코드, 컴포넌트 |
| 백엔드 개발자 | 데이터 요구사항, 지표 정의 | REST/GraphQL API, 데이터 모델 |
| 데이터 엔지니어/애널리스트 | 원천 데이터, KPI 정의 | 정제된 데이터셋, 지표 계산 로직 |
| QA 엔지니어 | 완성된 기능, 테스트 케이스 | 테스트 리포트, 버그 티켓 |
| DevOps 엔지니어 | 코드베이스, 환경 요구사항 | 로컬 실행 환경, CI/CD 파이프라인 |

---

### 3. 사람의 승인 지점
- **요구사항 승인**: PM이 작성한 요구사항 문서 검토  
- **디자인 승인**: UX/UI 시안 확정 시 이해관계자 승인  
- **개발 완료 승인**: 주요 기능 구현 후 코드 리뷰 및 머지 승인  
- **테스트 승인**: QA 결과 검토 후 배포 전 승인  

---

### 4. 로컬에서 검증할 명령
- **프론트엔드**:  
  - `npm run dev` → 로컬 서버 실행  
  - `npm run lint` → 코드 스타일 검증  
  - `npm run test` → 단위 테스트 실행  
- **백엔드**:  
  - `pytest` 또는 `npm run test` → API 단위 테스트  
  - `docker-compose up` → 로컬 환경에서 서비스 실행  
- **데이터**:  
  - SQL 쿼리 검증, 데이터 샘플링 테스트  
- **DevOps**:  
  - `docker build .` → 이미지 빌드 검증  
  - `kubectl apply -f ...` (로컬 클러스터) → 배포 테스트
